ASSPIS 项目文件说明（对接简版）

本说明用于帮助前后端/数据库对接人员快速理解当前目录下各文件的大致作用。
以下说明以“运行服务所需模块 / 离线评估与实验模块 / 示例数据与输出文件”进行划分，便于区分哪些是部署时重点关注的，哪些主要用于实验与验证。

一、运行服务所需模块（对接重点）

1. main.py
- 后端主入口文件。
- 一般负责启动接口服务，并将前端请求分发到预测、雷达图、历史数据读取等功能模块。
- 前端对接时，通常优先关注本文件中暴露的接口路径、请求参数、返回 JSON 结构。

2. model.py
- 核心预测模块。
- 负责销量预测相关的主要逻辑，通常包括特征组织、模型加载/训练、预测计算、what-if 分析等核心能力。
- 如果后续要替换模型、调整输入字段、修改预测口径，通常优先看这个文件。

3. quantile_forecast.py
- 分位数预测模块。
- 负责输出类似 P10 / P50 / P90 这类区间预测结果，用于展示预测范围、不确定性或上下界。
- 若前端需要展示“置信区间”“预测带”或“高/中/低情景”，通常与本文件相关。

4. radar.py
- 雷达图业务模块。
- 负责根据业务维度计算各项能力分数，并生成五力/多维能力雷达图所需的数据结构。
- 若后续从 Excel 切换到数据库读取，通常需要优先修改本文件中的数据读取部分。

5. dealer_data_preprocessing.py
- 数据预处理模块。
- 负责原始经销商数据的清洗、整理、字段标准化、结构转换等，为预测模块和图表模块提供可直接使用的数据。
- 数据源变更时（Excel → 数据库 / 新字段接入），此文件通常是重点修改位置之一。

6. selector_service.py
- 模型选择与部署产物生成模块。
- 主要用于离线阶段完成候选模型比较、最优方案选择，并输出上线所需的 bundle / manifest / registry 等文件。
- 正式部署时，在线服务一般直接读取其产物，而不是每次重新做大规模搜索。

7. line_chart.py
- 折线图数据或折线图生成模块。
- 用于展示销量或其他经营维度的时间序列变化。
- 若前端已自行绘图，则此文件更多是提供折线图所需数据；若后端直接生成图，也可能由此文件负责。

二、离线评估与实验模块（通常不直接给前端调用）

8. eval_phase2.py
- 第二阶段实验评估脚本。
- 主要用于离线评估模型效果，如滚动验证、误差统计、结果对比等。
- 更偏研究/实验用途，不属于前端联调主链路。

9. eval5.5.py
- 特定实验章节或特定评估口径的评估脚本。
- 通常用于生成论文/报告中的对比结果、指标表和验证结论。
- 一般不作为线上接口直接依赖。

10. eval_all_dealers_backtest_api.py
- 面向全经销商的批量回测/接口评估脚本。
- 可能用于批量检查多个经销商在某套接口或预测逻辑下的表现。
- 主要用于验证和回归测试。

11. backtest_quantiles_offline.py
- 分位数预测的离线回测脚本。
- 用于验证 quantile forecast 的历史表现是否合理，例如区间宽度、覆盖情况等。

12. coverage_report.py
- 覆盖率统计脚本。
- 主要用于检验预测区间是否覆盖真实值，并输出覆盖率报告。
- 常与分位数预测评估联动使用。

13. audit_dealers_completeness.py
- 经销商数据完整性审计脚本。
- 用于检查哪些经销商月份齐全、哪些字段缺失较多，便于筛选可用于建模或评估的样本。

14. run_experiment.py
- 实验启动脚本或实验流程封装脚本。
- 用于简化离线实验执行，不参与线上服务。

15. smoke_test_quantiles_api.py
- 分位数预测接口的冒烟测试脚本。
- 用于快速确认接口是否能正常启动、返回结构是否正确，适合部署后做基础连通性检查。

三、示例数据、配置输出与静态结果

16. 2024test.xlsx
- 当前使用的测试输入样例文件。
- 在数据库未接入前，可作为临时输入源使用。
- 后续若改为数据库读取，此文件可保留为调试样例，不再作为正式数据源。

17. quantified_scores_with_monthly_forces_*.xlsx
- 雷达图/五力分析相关的中间结果或输入结果文件。
- 一般存放每月的能力分数、五力指标等数据。
- 后续若改为数据库存储，这类 Excel 文件通常会被数据库表替代。

18. B440021_radar_chart_year.html
- 雷达图静态输出示例。
- 主要用于查看图表生成效果，不属于核心逻辑代码。

四、对接时建议优先关注的链路

若是前端/数据库对接，建议按下面顺序理解项目：
1）main.py：先看接口入口与返回结构；
2）model.py / quantile_forecast.py：看预测逻辑与输入输出；
3）radar.py：看雷达图数据结构；
4）dealer_data_preprocessing.py：看源数据字段如何进入模型；
5）selector_service.py：看部署产物如何生成与加载。

五、可部署链路和实验链路
1.部署链路重点保留：
- main.py
- model.py
- quantile_forecast.py
- radar.py
- dealer_data_preprocessing.py
- selector_service.py
- 运行时实际依赖的数据文件 / bundle / manifest / registry

2. 实验分析：
- eval_phase2.py
- eval5.5.py
- eval_all_dealers_backtest_api.py
- backtest_quantiles_offline.py
- coverage_report.py
- audit_dealers_completeness.py
- run_experiment.py
- smoke_test_quantiles_api.py

六、当前目录的整体理解

可以将当前项目粗略理解为三层：
- 数据层：Excel / 预处理脚本 / 完整性检查脚本
- 模型层：model.py、quantile_forecast.py、selector_service.py
- 服务与展示层：main.py、radar.py、line_chart.py、HTML 输出文件

如果后续要进行数据库改造，通常优先替换的是“数据读取入口”，而不是整套预测逻辑本身。
如果后续要与前端联调，最关键的是先固定接口字段、返回格式和图表所需的数据结构。
